# ADR-002: Secrets Management — SOPS + GPG

**Status:** Accepted (Active)

**Date:** 2026-05-04

**Authors:** Matei Rimboaca

---

## Context

A GitOps workflow stores all infrastructure configuration in Git. However, secrets (API keys, passwords, credentials) cannot be stored unencrypted in version control.

This homelab requires:
- Secrets stored in Git (for GitOps integrity)
- Secrets encrypted (never plaintext)
- Easy decryption for deployment
- Decryption key stays with the user (not stored in Kubernetes)
- Contributors can fork the repo and generate their own secrets

Multiple secret management approaches exist:
- **SOPS + GPG/AWS-KMS** — File-level encryption, portable
- **Sealed Secrets** — Kubernetes-bound secrets, cluster-specific
- **External Secrets Operator** — Fetch from external vault
- **Bitnami Sealed Secrets** — Similar to above
- **Hashicorp Vault** — Full-featured secret management system
- **Plain Kubernetes Secrets** — Unencrypted (not acceptable)

---

## Decision

**Use SOPS (Secrets Operations) with GPG encryption:**
- Secrets are SOPS-encrypted in Git
- Each user has their own GPG key
- `setup.sh` configures SOPS with the user's GPG key fingerprint
- ArgoCD + AVP (Argocd Vault Plugin) decrypts secrets during deployment
- Contributors can fork the repo and use their own GPG keys

---

## Rationale

### Why SOPS + GPG over Sealed Secrets

1. **Portability**
   - SOPS is framework-agnostic (works with any Kubernetes, cloud provider, even non-K8s)
   - Sealed Secrets tie encryption keys to a specific cluster
   - If you rebuild the cluster, sealed secrets become unrecoverable

2. **Key management**
   - SOPS uses your personal GPG key (likely already on your system)
   - No need to back up cluster secrets separately
   - You control who can decrypt

3. **Contributor-friendly**
   - New contributors can fork the repo
   - Run `setup.sh` with their own GPG key
   - Secrets re-encrypt automatically for their environment
   - No shared cluster secrets needed

4. **Git history**
   - Encrypted values are opaque in git diff
   - `sops` only decrypts what you ask for
   - Smaller, cleaner git history than Sealed Secrets

5. **ArgoCD integration**
   - ArgoCD has native AVP support
   - Decryption happens in ArgoCD, not in Kubernetes
   - Secrets are never plaintext in etcd

### Why not External Secrets Operator (ESO)

- **Complexity:** Requires a separate secret backend (Vault, AWS Secrets Manager, etc.)
- **Dependency:** Adds operational burden (backup, high availability, access control)
- **Overkill:** Homela environment doesn't need Vault-level features

### Why not Sealed Secrets

- **Non-portable:** Secrets are bound to cluster encryption key
- **Less flexible:** Contributors can't easily generate their own secrets
- **Recovery risk:** Losing cluster key = losing all secrets

## Implementation

### SOPS Configuration (.sops.yaml)

```yaml
creation_rules:
  - path_regex: .*secret.*\.ya?ml$
    encrypted: true
    pgp: <YOUR_GPG_KEY_FINGERPRINT>
  - encrypted: false
```

Generated by `setup.sh` with the user's GPG key.

### Secret File Format (cluster/infrastructure/*/secret.yaml)

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-secret
  namespace: my-namespace
stringData:
  api_key: actual-encrypted-value
  password: another-encrypted-value
# sops:
#   pgp:
#     - created_at: 2026-05-04T...
#       enc: |-
#         -----BEGIN PGP MESSAGE-----
#         ...
#         -----END PGP MESSAGE-----
```

The entire Secret is encrypted by SOPS (all values, not just specific fields).

### ArgoCD + AVP Decryption

ArgoCD config manager plugin:
1. Detects files containing encrypted values (sops metadata)
2. Runs `sops -d <file>` to decrypt
3. Passes decrypted YAML to kubectl apply
4. Kubernetes receives plaintext Secret

Decryption happens in ArgoCD (not Kubernetes), so secrets never hit etcd unencrypted.

### Encrypting New Secrets

```bash
# 1. Create secret file (with plaintext)
cat > cluster/infrastructure/my-app/secret.yaml << 'EOF'
apiVersion: v1
kind: Secret
metadata:
  name: my-secret
  namespace: my-app
stringData:
  api_key: REPLACE_WITH_API_KEY
EOF

# 2. Encrypt with SOPS
sops --encrypt --in-place cluster/infrastructure/my-app/secret.yaml

# 3. Commit to Git
git add cluster/infrastructure/my-app/secret.yaml
git commit -m "feat: add my-app secret"
```

## Consequences

### Positive

✅ **Portable:** Secrets travel with your infrastructure code
✅ **Contributor-friendly:** Easy to fork and use your own keys
✅ **Git-native:** All config in one place (no external systems)
✅ **Simple:** Single tool (sops), easy to understand
✅ **Auditable:** Git log shows when secrets changed (not the values)
✅ **Flexible:** Works with any cluster, any cloud, even non-K8s

### Negative

❌ **GPG complexity:** GPG key management has a learning curve
❌ **Key loss:** Losing your GPG private key = losing encrypted secrets
❌ **No secret rotation in-place:** Requires re-encryption + git commit
❌ **Manual re-encryption:** If you want to share secrets with contributors, they need their own keys

### Mitigations

- Document GPG key backup (cluster/docs/SECRETS-ROTATION.md)
- Provide templates for common secrets (cluster/infrastructure/*/secret.yaml)
- Use `setup.sh` to automate SOPS configuration
- ArgoCD AVP sidecar handles decryption transparently

## Alternatives Considered

### Sealed Secrets (Bitnami)
- Cluster-bound encryption
- Portable only within same cluster
- Better for multi-cluster scenarios (but overkill here)

### External Secrets Operator (ESO)
- Integrates with Vault, AWS Secrets Manager, etc.
- Adds operational complexity
- Good for enterprise (not homelab)

### Hashicorp Vault
- Full-featured secret management
- Requires Vault infrastructure
- Overkill for homelab

### Plain Kubernetes Secrets (unencrypted)
- ❌ Unsafe
- Anyone with etcd access reads all secrets
- Not acceptable for this repo
